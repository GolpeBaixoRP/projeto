1. Objetivo do Projeto

O projeto Meu Sonho em um Clique tem como objetivo implementar um sistema determinístico de preparação de mídia removível, projetado para:

preparar discos USB

reconstruir layout de partição

padronizar estrutura de armazenamento

aplicar formatação controlada

validar o resultado final

O sistema foi projetado com filosofia semelhante a storage appliances, onde a mídia é preparada segundo uma política definida e verificada antes de ser considerada pronta.

A política principal definida para o sistema é:

discos alvo devem ser convertidos para MBR

o sistema deve suportar FAT32 e exFAT

o resultado deve ser verificado antes da liberação da unidade

2. Arquitetura do Sistema

O sistema foi estruturado em camadas separadas para manter isolamento entre:

3. Pipeline Operacional

Cada operação de formatação executa o seguinte pipeline:

INIT / lock
SNAPSHOT refresh + identidade
PRECHECK guardian
IPC start
Worker execution
IPC end
Snapshot refresh pós
Verify evidência
Commit / release

Descrição das etapas:

INIT / lock

Garante exclusividade de operação sobre o disco selecionado.

SNAPSHOT refresh

Coleta estado atual do hardware através das APIs do Windows.

PRECHECK guardian

Valida se o disco pode ser manipulado.

Exemplos de bloqueio:

disco do sistema

disco interno

estado inconsistente

IPC start

Disparo do worker PowerShell responsável pela manipulação real da mídia.

Worker execution

Executa:

clean
convert mbr
create partition primary
format
assign

Snapshot refresh pós

Reconsulta o estado físico após a operação.

Verify evidência

Valida:

estilo da partição

sistema de arquivos

cluster size

presença de letra de unidade

Commit

Libera a unidade e marca o disco como READY.

4. Política de Preparação de Mídia

A política implementada pelo sistema define que discos preparados devem seguir:

Partition Style : MBR
Filesystem      : FAT32 ou exFAT
Cluster size    : definido pelo worker
Volume letter   : atribuída

Essa política garante compatibilidade com:

OPL

PlayStation 2

sistemas legados

ferramentas que exigem MBR

2. Consolidação da política funcional
2.1 Definir política final de filesystem

Hoje já está claro:

FAT32

obrigatório:

MBR

BlockSize = 32768

exFAT

Aqui você precisa decidir formalmente se a política é:

opção A — permissiva

MBR

exFAT

cluster pode variar

opção B — determinística

MBR

exFAT

BlockSize = 32768

Pelo espírito do projeto, eu recomendo opção B.
Porque o seu sistema não quer “deixar o Windows decidir”. Ele quer impor política.

Se escolher isso, o verify também deve cobrar 32768 em exFAT.

2.2 Formalizar estado READY

O disco só deve virar READY quando estas condições forem verdadeiras:

PartitionStyle == MBR

existe exatamente uma partição principal operacional

existe letra de unidade

filesystem bate com o pedido

cluster size bate com a política

worker retornou sucesso

verify passou

Se qualquer uma falhar, o estado deve continuar:

NEEDS_PREPARATION

ou ERROR no futuro, se você quiser criar esse estado

3. Consolidação arquitetural
3.1 Preservar single-thread

Seu projeto deve continuar single-thread na operação principal.

Regra

uma operação por vez

um disco selecionado por vez

um worker por vez

nenhum processamento concorrente de múltiplos discos

Isso evita:

race condition

conflito de letra

disputa de DiskPart

estados fantasmas do Windows

3.2 Preservar hierarquia rígida

A ordem de autoridade deve continuar assim:

main.py
↓
OperationController
↓
Guardian / Policy
↓
RealFormatService
↓
PowerShell runner
↓
Worker PS1
Regra

main.py não decide regra de mídia

worker PS1 não decide política global

controller continua sendo o soberano do pipeline

Isso é importante porque o worker é “mão”, não “cérebro”.

3.3 Proibir lógica crítica espalhada

Toda política crítica deve ter dono explícito:

Controller

orquestração

progresso

commit/fail

verificação final

Guardian

identidade física

proteção de dispositivo

Worker

execução concreta de disco

Service

ponte de IPC / captura / parse

Se a mesma regra existir em dois lugares, você cria uma seita e não um sistema.

5. Consolidação do runner
5.1 Endurecer utils/powershell_runner.py

Esse arquivo precisa ficar blindado porque ele é o túnel entre Python e PowerShell.

processo oculto

sem popup

kill da árvore em KeyboardInterrupt

Se isso não estiver feito, um worker pode virar zumbi e o operador fica refém do terminal. Coisa deselegante, feia e bem Windows.
